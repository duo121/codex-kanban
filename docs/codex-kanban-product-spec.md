# Codex Kanban 产品文档

## 1. 文档信息

- 文档类型：产品需求文档 + 一期技术方案
- 产品名称：codex-kanban
- 当前定位：基于官方 Codex 仓库做长期维护的上游跟随型增强产品
- 版本范围：MVP v1

## 2. 背景与问题

当前官方 Codex CLI 的主路径是“一个窗口聚焦一个主会话”。

当用户需要同时推进多个任务时，通常需要同时打开大量终端窗口。例如：

- 25 个并行任务，需要打开 25 个 Codex 窗口
- 用户需要频繁在多个终端窗口之间切换
- 每个窗口的 UI、滚动位置、任务状态都是割裂的
- 机器上会堆积大量终端实例，带来额外的内存和管理成本

目标不是重做一套新的 TUI，也不是替换官方聊天区，而是在尽量复用官方 UI 的前提下，把“多个会话的管理能力”收口到更少的 Codex 窗口中。

期望使用方式：

- 用户可以同时开 5 个 Codex 窗口
- 每个窗口绑定 1 个看板
- 每个看板管理 5 个会话
- 总共只需要 5 个窗口，就能管理 25 个会话

## 3. 产品目标

### 3.1 核心目标

1. 尽可能复用官方 Codex UI 和交互组件。
2. 在单个 Codex 窗口内，快速管理多个会话。
3. 将多个会话组织到“看板”中，降低多终端窗口数量。
4. 支持后台并发执行多个会话，并在当前窗口查看会话状态。
5. 降低后续同步官方版本时的维护成本。

### 3.2 设计原则

1. 右侧聊天区保持官方 Codex 交互方式，不新造一套主界面。
2. 看板和会话管理优先复用官方现有选择器弹层，而不是引入常驻侧栏。
3. “看板”与“真实聊天会话”解耦。
4. `/resume` 继续保留官方语义，不改成看板入口。
5. 新功能以“附加层”方式叠加在官方结构上，避免重写主状态机。

## 4. 非目标

以下内容不纳入 MVP v1：

1. 不做分组内再分组。
2. 不做类似 Jira/Trello 的拖拽列式可视化看板。
3. 不在 v1 中实现窗口关闭后后台会话继续运行的守护进程模式。
4. 不修改官方 `/resume` 的全局历史会话语义。
5. 不在 v1 中做大规模品牌替换，不全局改 crate 名称、包名、二进制名。

## 5. 目标用户与典型场景

### 5.1 目标用户

- 高频使用 Codex CLI 的开发者
- 需要并行推进多个子任务的用户
- 经常同时开多个终端窗口管理多个 Codex 会话的用户

### 5.2 典型场景

#### 场景 A：单看板多会话并发

用户正在处理“支付系统重构”任务，希望在一个窗口内同时推进：

- 会话 1：主实现
- 会话 2：测试修复
- 会话 3：日志排查
- 会话 4：文档整理
- 会话 5：性能分析

用户把这 5 个会话放进同一个看板，在一个窗口里切换和跟进。

#### 场景 B：多窗口多看板

用户同时推进 5 个大任务：

- 任务 A：支付重构
- 任务 B：登录排障
- 任务 C：数据同步
- 任务 D：发布保障
- 任务 E：前端优化

每个大任务对应一个看板窗口，每个看板窗口内再管理多个会话。

#### 场景 C：窗口意外关闭后恢复

用户误关了一个绑定看板的窗口。下次重新打开 Codex 后，通过 `/kb` 找回之前的看板，再进入会话列表，继续管理该看板中的会话。

## 6. 核心概念

### 6.1 看板（Board）

看板是用户定义的“当前窗口内的会话工作集”。

看板本身不存储聊天正文，只存储：

- 看板名称
- 看板下包含哪些会话
- 会话在看板中的排序
- 会话在看板中的状态快照
- 最近激活的会话

### 6.2 会话（Board Session）

看板中的会话是对真实 Codex 会话的引用，不是新的一层聊天存储。

### 6.3 真实会话（Underlying Thread / Session）

真实会话仍由官方 Codex 原有机制管理和持久化。看板只是组织和切换这些会话的产品层能力。

### 6.4 看板窗口（Board-bound Window）

当一个 Codex 窗口绑定某个看板后，该窗口进入“看板窗口模式”：

- `/kb` 可继续管理看板
- `~` 直接打开当前看板的会话列表
- 新建会话默认加入当前看板

### 6.5 普通窗口（Unbound Window）

未绑定看板时，窗口保持官方单会话模式。此时按 `~` 不做看板会话切换，而是提示用户先通过 `/kb` 选择或创建看板。

## 7. 信息架构

```text
Codex Kanban
├── 全局看板列表
│   ├── 看板 A
│   │   ├── 会话 1
│   │   ├── 会话 2
│   │   └── 会话 3
│   ├── 看板 B
│   └── 看板 C
└── 官方全局历史会话（/resume，保持原语义）
```

关键边界：

- `/kb` 管理的是“看板”
- `~` 管理的是“当前看板中的会话”
- `/resume` 管理的是“官方全局历史会话”

三者必须严格分开。

### 8.1 `/kb` 打开的看板列表

```text

│ 搜索看板…                                         │
│                                                   │

│   feat: 发布保障         4 会话  1 运行中  2 待跟进 │
│                                                   │
│ Enter 进入  n 新建  r 重命名  d 删除  Esc 关闭      │
└───────────────────────────────────────────────────┘
```

### 8.2 `~` 打开的当前看板会话列表

```text
┌ 当前看板会话 ──────────────────────────────────────┐
│ 搜索会话…                                         │
│                                                   │
│ > feat: 主实现                运行中      当前      │
│   feat: 测试修复              待跟进                │
│   feat: 日志排查              等审批                │
│   feat: 文档整理              已跟进                │
│   feat: 性能分析              异常                  │
│                                                   │
│ Enter 切换  n 新建  r 重命名  d 移出看板  Esc 返回  │
└───────────────────────────────────────────────────┘
```

### 8.3 右侧聊天区

右侧聊天区保持官方 Codex 样式与行为，不引入新的常驻左侧栏。

原因：

1. 复用官方 UI 成本最低。
2. 不会因为 transcript 滚动导致左侧看板消失或布局错位。
3. 后续同步官方 TUI 更新时冲突最小。

## 9. 交互逻辑

### 9.1 `/kb`：全局看板入口

#### 功能

- 查看所有看板
- 搜索看板
- 新增看板
- 重命名看板
- 删除看板
- 进入看板

#### 操作

- 输入 `/kb`：打开看板列表
- 输入文字：过滤看板
- `↑` / `↓`：选择看板
- `n`：新增看板
- `r`：重命名当前看板
- `d`：删除当前看板，需二次确认
- `Enter`：当前窗口绑定该看板，并进入该看板的会话列表
- `Esc`：关闭看板列表

### 9.2 `~`：当前看板会话快捷入口

#### 触发条件

- 当前窗口已绑定看板
- 输入框为空时按 `~`

#### 行为

- 直接打开当前看板的会话列表
- 不再经过全局看板列表
- 这是高频切换入口

#### 未绑定看板时

- 不打开空白会话列表
- 给出提示：请先使用 `/kb` 选择或创建看板

### 9.3 看板内会话管理

#### 操作

- 输入文字：过滤会话
- `↑` / `↓`：选择会话
- `Enter`：切换到该会话
- `n`：新建会话并加入当前看板
- `r`：重命名会话
- `d`：将会话移出当前看板，需二次确认
- `Esc`：关闭会话列表，返回聊天区

### 9.4 当前会话被移出看板时的行为

如果用户把当前激活会话移出当前看板：

1. 若看板中还有其他会话，自动切到下一个会话。
2. 若看板中已无其他会话，自动新建一个会话并加入看板。

原则：窗口内始终保持一个可用的当前会话，不允许出现“绑定了看板但没有活跃会话”的状态。

### 9.5 删除看板时的行为

当用户删除当前窗口正在绑定的看板：

1. 需要二次确认。
2. 删除后，该窗口解除绑定状态。
3. 当前聊天区先保持不动，退化为普通窗口模式。
4. 下次按 `~` 时，提示先通过 `/kb` 绑定看板。

### 9.6 窗口关闭后的恢复

如果用户关闭了绑定看板的窗口：

1. 看板元数据保留。
2. 会话历史仍可通过原有持久化恢复。
3. 下次打开 Codex 后，用 `/kb` 找回目标看板。
4. 进入看板后，再选择该看板中的会话。

注意：MVP v1 不保证窗口关闭后后台会话继续运行。

## 10. 后台并发与状态

### 10.1 并发模型

单个看板窗口内的多个会话可以并发进行。

更准确地说：

- 它们运行在同一个 Codex 进程中
- 属于多个后台 thread / task
- 当前激活会话负责显示在右侧聊天区
- 非激活会话继续在后台积累事件和状态

### 10.2 对主程序的影响

目标是“不阻塞主 TUI 交互”，而不是“完全没有资源占用”。

意味着：

- UI 仍应保持可切换、可输入、可查看状态
- 但多个会话仍共享本机 CPU、内存、网络和本地工具执行资源

### 10.3 状态定义

MVP 建议支持以下状态：

- `运行中`：会话正在生成回复或执行任务
- `待跟进`：会话在非激活状态下产生了新输出
- `已跟进`：用户已经切回并查看过最新输出
- `等审批`：会话正在等待用户审批
- `异常`：会话 attach 失败、执行失败或退出异常

### 10.4 状态流转

```text
新建 -> 运行中 -> 待跟进 -> 已跟进
                 └-> 等审批
                 └-> 异常
```

规则：

- 非当前会话如果产生新输出，状态转为 `待跟进`
- 用户切换回该会话并完成查看后，状态转为 `已跟进`
- 会话正在执行时显示 `运行中`
- 会话等待用户输入审批时显示 `等审批`

## 11. 产品边界与语义约束

### 11.1 会话移出看板 != 删除真实会话

`d` 在会话列表中的语义必须是：

- 从当前看板移除该会话
- 不删除真实聊天记录
- 不删除 rollout 文件
- 不影响 `/resume` 中的历史会话

### 11.2 删除看板 != 删除其中所有会话

删除看板只删除：

- 看板本身
- 看板与会话的映射关系

不删除：

- 会话正文
- 会话持久化文件
- 官方历史恢复能力

### 11.3 `/resume` 保持原语义

`/resume` 仍然用于官方全局历史会话恢复。

它不负责：

- 看板管理
- 当前看板会话切换
- 当前窗口工作集筛选

## 12. 技术方案

### 12.1 总体方案

采用“官方聊天区 + 官方选择器弹层 + 新增看板元数据层”的方式实现。

```text
官方聊天区（复用）
  +
官方 ListSelectionView 弹层（复用）
  +
Board Registry / Board Session Registry（新增）
  +
线程切换与后台事件存储（复用官方 thread 模型）
```

### 12.2 UI 复用策略

优先复用官方现有选择器组件：

- 看板列表：一个选择器弹层
- 当前看板会话列表：另一个选择器弹层
- 重命名：复用官方现有 rename prompt
- 删除确认：复用现有确认弹层模式，或做一层可复用确认视图

不引入新的 TUI 框架，不增加常驻左侧 pane。

### 12.3 会话切换策略

不复用官方 `/resume` 路径进行当前窗口内多会话切换。

原因：

- `/resume` 的产品语义是“恢复全局历史会话”
- `/resume` 当前实现更偏向替换当前会话
- 不适合当前窗口内的高频多会话并发切换

正确做法：

- 复用官方已有的 thread 切换模型
- 当前窗口维护多个活跃 thread 的状态存储
- 切换时只更换 active thread
- 非激活会话继续在后台接收事件

### 12.4 持久化策略

看板信息与真实会话正文分开存储。

建议新增单独的看板持久化存储，用于保存：

- 看板列表
- 看板名称
- 看板与会话的映射关系
- 会话排序
- 最近激活会话
- 用户已读/待跟进状态

建议位置：

- 放在 `codex_home` 下
- 使用单独 SQLite 文件或现有数据库中的独立表

MVP 推荐：SQLite。

原因：

1. 支持多窗口并发读写。
2. 比 JSON 更适合后续扩展。
3. 易于维护会话状态、排序和时间戳。

### 12.5 建议数据模型

#### boards

- `id`
- `name`
- `created_at`
- `updated_at`
- `last_selected_thread_id`

#### board_sessions

- `id`
- `board_id`
- `thread_id`
- `title_snapshot`
- `sort_order`
- `status`
- `last_seen_event_idx`
- `last_event_idx`
- `added_at`
- `removed_at`

说明：

- `removed_at` 非空表示“从看板移除”，不是删除真实会话
- `title_snapshot` 仅作看板列表快速展示，真实标题仍以线程数据为准

### 12.6 窗口绑定状态

窗口级只维护一个运行时状态：

- `bound_board_id: Option<BoardId>`

规则：

- 有值：说明当前窗口绑定某个看板
- 无值：当前窗口是普通窗口模式

这个状态可以只做运行时内存态，不需要持久化成“窗口实例”。

### 12.7 新会话加入规则

在绑定看板的窗口中：

- 用户通过 `n` 新建会话
- 或通过原有 `/new` 新建会话

系统都应默认把该新会话加入当前看板。

### 12.8 右侧聊天区显示策略

切换会话时采用：

- 清屏
- 重放目标会话
- 展示该会话上下文

不采用：

- 把新会话内容直接接在当前屏幕下面继续追加

原因：

1. 与官方 thread 切换行为一致
2. 用户认知更稳定
3. 不会混淆两个会话的 transcript

## 13. 与官方仓库的协同维护策略

### 13.1 仓库策略

当前仓库维护方式：

- `origin`：用户 fork 仓库
- `upstream`：官方 `openai/codex`

### 13.2 同步原则

1. 尽量少改官方聊天区主结构。
2. 尽量少动 `/resume`、主输入框、核心渲染路径。
3. 新增功能尽量通过独立模块和最小接线方式接入。
4. 看板能力优先放在新的模块层，而不是散落到大量现有文件中。

### 13.3 命名与品牌策略

MVP 阶段：

- 仓库目录名使用 `codex-kanban`
- 产品文档中使用 `codex-kanban`
- 不立即全局替换官方项目内部名称

这样可以降低与 upstream 的合并冲突。

## 14. 风险与限制

### 14.1 窗口关闭导致后台任务终止

MVP v1 中，会话并发执行依附于当前 Codex 进程。

因此：

- 关闭该窗口
- 杀掉该 Codex 进程

都可能终止该窗口内仍在运行的后台会话。

这是已知限制，不作为 v1 阻断项。

### 14.2 多会话共享资源

单窗口内并发会话会共享：

- CPU
- 内存
- 网络
- 本地工具调用资源

因此“更少的终端窗口”不等于“更少的任务开销”，只是显著减少了终端实例管理成本。

### 14.3 状态一致性

如果用户同时开多个看板窗口，并且某个真实会话被加入多个看板，需要清晰定义状态展示策略。

MVP 建议：

- 允许一个真实会话属于多个看板
- 待跟进/已跟进状态按“看板视角”记录
- 运行中/等审批状态按“真实会话视角”计算

## 15. MVP 范围

### 15.1 必做

1. `/kb` 看板列表
2. 看板增删改查
3. `Enter` 进入看板
4. `~` 打开当前看板会话列表
5. 当前看板会话搜索、切换、新建、重命名、移出
6. 当前会话移出后的自动补位逻辑
7. 看板与会话映射持久化
8. 会话基础状态展示：运行中 / 待跟进 / 已跟进 / 等审批 / 异常

### 15.2 可选增强

1. 看板列表展示统计信息
2. 最近使用的看板排序
3. 当前看板默认会话恢复
4. 会话列表中的描述区展示最后一条摘要或最近更新时间

### 15.3 暂不做

1. 常驻左侧看板
2. 多级分组
3. 拖拽排序
4. 进程外守护执行
5. 跨机器同步看板

## 16. 里程碑建议

### 里程碑 1：看板骨架

- 建立看板存储
- `/kb` 看板 CRUD
- 窗口绑定/解绑看板

### 里程碑 2：当前看板会话列表

- `~` 打开当前看板会话列表
- 新建/切换/移出/重命名会话
- 列表过滤能力

### 里程碑 3：状态系统

- 后台运行状态
- 待跟进/已跟进
- 等审批/异常

### 里程碑 4：稳定性与恢复

- 窗口关闭后的看板恢复
- 边界条件处理
- 文档补全与回归测试

## 17. 验收标准

产品达到 MVP 可用标准，需满足：

1. 用户可以通过 `/kb` 创建、重命名、删除、进入看板。
2. 进入看板后，用户可以通过 `~` 快速打开该看板的会话列表。
3. 用户可以在同一窗口中管理多个会话，而不需要再为每个会话单独开窗口。
4. 非当前会话的后台状态可以在会话列表中看到。
5. 将会话移出看板不会删除真实聊天记录。
6. 关闭窗口后，看板和会话关系仍能通过 `/kb` 找回。
7. 右侧聊天区依旧保持官方 Codex 使用体验。
8. 新功能引入后，不破坏官方 `/resume` 语义。

## 18. 一句话产品定义

codex-kanban 是一个基于官方 Codex 持续演进的轻量增强版本：在不重写官方聊天 UI 的前提下，引入“看板 + 当前窗口多会话管理”能力，用更少的终端窗口管理更多并发任务。
